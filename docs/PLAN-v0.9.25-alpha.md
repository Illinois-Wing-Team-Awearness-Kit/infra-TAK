> **Status:** DRAFT

## Why v0.9.25 exists

v0.9.x added Phase 2 container hardening (CVE-2026-31431 / Copy Fail mitigation) to Authentik's `server` and `ldap` services. The injector at `app.py:43556-43560` uses a **whole-file literal-substring** check for idempotency:

```python
for _old, _new in (
    ('command: server\n', 'cap_drop:\n      - ALL\n    security_opt:\n      - no-new-privileges:true\n    command: server\n'),
):
    if _old in _ak and _new not in _ak:
        _ak = _ak.replace(_old, _new, 1)
```

If the operator's `~/authentik/docker-compose.yml` was generated by an earlier release whose template placed `cap_drop:` in a *slightly* different position (any whitespace or line-ordering drift between the cap_drop block and `command: server`), the literal `_new` substring is **not** found anywhere in the file, the guard returns False, and the hardening pass **inserts a second `cap_drop:` block** right above `command: server`. The resulting YAML has two `cap_drop:` mapping keys in the same `server:` service. Compose v2's Go YAML parser refuses to load the file at all.

Downstream impact in `_set_webadmin_password()` (`app.py:33663-33667`):

```python
elif os.path.exists(os.path.expanduser('~/authentik/docker-compose.yml')):
    r = subprocess.run('cd ~/authentik && docker compose up -d --force-recreate ldap 2>&1',
        shell=True, capture_output=True, text=True, timeout=90)
    if r.returncode != 0:
        return False, 'Password set but LDAP restart failed: ' + (r.stderr or r.stdout or '')[:120]
```

The Authentik API call to set the password **succeeds** (so login *eventually* works against the LDAP outpost's cache window), but the `docker compose up -d --force-recreate ldap` call that's supposed to flush the LDAP outpost's bind cache **fails on YAML parse**. Result: clicking *Sync webadmin* shows a red banner, 8446 keeps binding against stale credentials, and the operator can't recover login from the console — they're forced to SSH into the Authentik host and hand-edit YAML.

The LDAP hardening at `app.py:43568-43581` is **not affected** because it uses a section-bounded check (`'cap_drop:' not in _ldap_block`). That's the pattern the server-section injector should have used from day one.

This release ships:
1. The actual code fix (per-service section-bounded check, mirroring the LDAP pattern).
2. A defensive duplicate-stripper to heal already-broken installs on the next Update Now (operators don't need to know there was a bug).
3. UI / error-surfacing improvements so when something *else* breaks the Authentik compose file in the future, the operator sees the full parse error and a one-line remediation hint instead of a 120-char truncation.
4. A pre-flight `docker compose config` validation on every Update Now, so future template patches that break YAML are caught before they're written.

## Item 1 — Per-service `cap_drop` hardening (idempotent, section-bounded)

### Problem

`app.py:43556-43560` uses a whole-file literal-substring guard. Drift in any part of the existing `server:` section relative to the literal causes a duplicate `cap_drop:` injection on the next Update Now.

### Goal

Mirror the LDAP pattern at `app.py:43568-43581`: locate the `server:` service block, bound it, and inject only if `cap_drop:` is not already present inside *that block*.

### Implementation plan

1. **Rewrite the `server:` injector** in `_auto_harden_containers()` (`app.py:43556-43560`):

   ```python
   # server: section-bounded check (anchor: '\n  server:\n', bound: next top-level service key or top-level 'volumes:' / 'networks:')
   _srv_start = _ak.find('\n  server:\n')
   if _srv_start != -1:
       import re as _re_harden
       _srv_end_m = _re_harden.search(r'\n  \w+:\n|\nvolumes:\n|\nnetworks:\n', _ak[_srv_start + 1:])
       _srv_end = _srv_start + 1 + (_srv_end_m.start() if _srv_end_m else len(_ak))
       _srv_block = _ak[_srv_start:_srv_end]
       if 'cap_drop:' not in _srv_block and 'restart: unless-stopped' in _srv_block:
           _srv_new = _srv_block.replace(
               '    restart: unless-stopped\n',
               '    restart: unless-stopped\n    cap_drop:\n      - ALL\n    security_opt:\n      - no-new-privileges:true\n',
               1
           )
           _ak = _ak[:_srv_start] + _srv_new + _ak[_srv_end:]
   ```

2. **Audit the rest of `_auto_harden_containers()`** for the same anti-pattern. Specifically:
   - The worker-block *removal* logic at `app.py:43562-43567` already uses literal-substring match on the *bad* state, which is safe (false negatives are OK there — the worst case is the legacy block stays, but worker is intentionally unhardened anyway, so that's a no-op visible only to inspection).
   - LDAP at `43568-43581` is already correct.
   - No other whole-file literal-block injections in the Authentik path.

3. **Audit Node-RED compose hardening** at `app.py:43472-43478` for the same issue:

   ```python
   if 'cap_drop:' not in content and 'container_name: nodered' in content:
       content = content.replace(
           'container_name: nodered\n',
           'container_name: nodered\n    user: "1000:1000"\n    cap_drop:\n      - ALL\n    security_opt:\n      - no-new-privileges:true\n    mem_limit: 2g\n',
           1
       )
   ```

   Node-RED's check uses whole-file `'cap_drop:' not in content` — which **is** correct here because Node-RED's compose only ever has one service (`nodered`). No section-bounding needed. Leave as-is, but add a code-comment explaining why this single-service case is safe.

## Item 2 — Defensive duplicate-`cap_drop` stripper

### Problem

Operators who already ran a v0.9.x Update Now that injected the duplicate `cap_drop:` block are stuck — their `~/authentik/docker-compose.yml` is unparseable, so the next Update Now's `_auto_harden_containers()` itself can't proceed past any step that calls `docker compose` (which is most of them). Without an explicit cleanup pass, Item 1 alone doesn't heal existing installs.

### Goal

Before any other hardening logic runs, detect and strip duplicate consecutive `cap_drop:` blocks inside each Authentik service section. This must be:
- Idempotent (re-running yields no change on a clean file).
- Safe (no false positives — must not strip the single legitimate `cap_drop:` block).
- Logged (operator sees `Post-update: Authentik — removed duplicate cap_drop in server section`).

### Implementation plan

1. **Add `_dedupe_authentik_capdrop(_ak: str) -> tuple[str, list[str]]`** as a new helper near `_auto_harden_containers()`. Returns the cleaned compose string and a list of section names that were de-duplicated.

2. **Algorithm** (per service block, anchored on `\n  <svc>:\n` for `server`, `worker`, `ldap`):
   - Find all positions of `    cap_drop:\n      - ALL\n    security_opt:\n      - no-new-privileges:true\n` inside the section block.
   - If count > 1, strip all but the first occurrence.
   - Also handle the bare `    cap_drop:\n      - ALL\n` (without security_opt) variant by walking adjacent `cap_drop:` mapping keys and keeping only the first.

3. **Call the helper at the start of `_auto_harden_containers()`**, before the `/var/run/docker.sock` strip:

   ```python
   _ak, _deduped = _dedupe_authentik_capdrop(_ak)
   for _svc in _deduped:
       print(f"Post-update: Authentik — removed duplicate cap_drop in {_svc} section")
   ```

4. **End-to-end smoke**: after the dedupe write but before `docker compose up -d --force-recreate worker server ldap`, call `docker compose -f ~/authentik/docker-compose.yml config >/dev/null` and refuse to proceed if it still fails (see Item 4). This catches *any* YAML breakage we caused or inherited, not just duplicate cap_drop.

## Item 3 — Surface the full compose-parse error to the operator

### Problem

`_set_webadmin_password()` truncates the error to 120 chars:

```python
return False, 'Password set but LDAP restart failed: ' + (r.stderr or r.stdout or '')[:120]
```

Field report shows the banner as:

> Password set but LDAP restart failed: failed to parse /root/authentik/docker-compose.yml: yaml: construct errors: line t line 44: mapping key 'cap_drop' al…

The truncation hides the line numbers of the *second* occurrence and the duplicated key name — the exact two pieces of information the operator needs to self-repair. Operators end up needing AI / dev help just to read the error.

### Goal

- Surface the full parse error (no 120-char cap) in the banner.
- If we can heuristically detect a duplicate-key parse error, append a one-line remediation hint.

### Implementation plan

1. **Bump truncation from 120 → 1000 chars** at `app.py:33667`. The console UI already wraps long error text; there's no reason to truncate here. Apply the same change at `app.py:33662` (remote LDAP restart error path) for consistency.

2. **Add a remediation hint** when the error looks like a duplicate-mapping-key parse failure. Detection: `'mapping key' in r.stderr and 'already defined' in r.stderr` (or fallback regex on `r.stdout`). Append:

   ```
   Likely cause: duplicate cap_drop or security_opt in ~/authentik/docker-compose.yml.
   Run "Update Now" — v0.9.25+ auto-strips duplicate blocks. Or SSH and remove the second cap_drop block manually.
   ```

3. **Also surface in the console UI** the underlying compose-validate command operators can run themselves:

   ```bash
   ssh root@<authentik-host> 'docker compose -f ~/authentik/docker-compose.yml config'
   ```

   Add this to the **TLS Auth & LDAP Sync** card description text (`app.py` HTML, near line `40892`).

## Item 4 — Pre-flight Authentik compose YAML validation on every Update Now

### Problem

`_auto_harden_containers()` writes `~/authentik/docker-compose.yml` and immediately calls `docker compose up -d --force-recreate worker server ldap`. If the write produced invalid YAML, the recreate fails silently inside a `subprocess.run(..., 2>/dev/null)` and the operator has no visibility — until the next Sync webadmin click, hours or days later, surfaces it.

### Goal

Validate that the compose file parses before we attempt any `docker compose up` that depends on it. If validation fails, log loudly and refuse to write.

### Implementation plan

1. **Add `_validate_authentik_compose(_ak: str) -> tuple[bool, str]`** as a new helper. It writes `_ak` to a temp file and runs `docker compose -f <tmp> config` (capture stderr). Returns `(True, '')` on success or `(False, stderr)` on failure.

2. **Before every write to `ak_compose`** in `_auto_harden_containers()` (and the remote deploy step at `app.py:25221-25224`), call the validator. On failure:
   - Log the full stderr with `Post-update: Authentik compose validation failed: <err>`.
   - Do NOT write the file.
   - Do NOT proceed to `docker compose up -d --force-recreate`.
   - Surface the failure in the console deploy banner / Updates log.

3. **One-time post-Item-2 dedupe verification**: after `_dedupe_authentik_capdrop()` strips duplicates, *also* validate the resulting string. If it still doesn't parse, log a distinct error so the operator knows the duplicate was symptomatic of something deeper (e.g. a leftover broken block from yet another release we haven't accounted for).

## Item 5 — Correct the misleading "Does not restart anything" UI text

### Problem

The console UI at `app.py:40896` describes the Sync webadmin button as:

> **Sync webadmin** — Only pushes the 8446 password from settings into Authentik. Does not restart anything.

But the code at `app.py:33660-33667` clearly recreates the `authentik-ldap-1` container as part of the sync flow (this is what's failing in the field report). Misleading documentation is its own bug — it directs operators away from the real root cause when they hit failures here.

### Goal

Match the description to the actual behavior.

### Implementation plan

1. **Update the description text** at `app.py:40896` to:

   > **Sync webadmin** — Pushes the 8446 password from settings into Authentik and recreates the LDAP outpost so the new password is honored immediately (flushes bind cache). Use after changing the webadmin password.

2. **Update the corresponding sentence below the Save password input** at `app.py:40905`:

   > Saving updates TAK Server flat-file (8446 login). Click **Sync webadmin** to also update Authentik and recreate the LDAP outpost.

## Acceptance criteria

- [ ] On a fresh Authentik install, `~/authentik/docker-compose.yml` contains exactly one `cap_drop:` block in `server:` and one in `ldap:` (none in `worker:` by design).
- [ ] On an Authentik install that **already** has a duplicate `cap_drop:` block from a prior release, the first Update Now after v0.9.25 strips it cleanly and logs `Post-update: Authentik — removed duplicate cap_drop in server section`.
- [ ] After dedupe, `docker compose -f ~/authentik/docker-compose.yml config` exits 0.
- [ ] Clicking **Sync webadmin to Authentik** on a freshly-deduped install succeeds end-to-end: password set in Authentik, LDAP outpost recreated, bind verification passes, no red banner.
- [ ] If a future bug re-introduces invalid YAML, the console banner shows the full parse error (not truncated at 120 chars) plus a one-line remediation hint.
- [ ] Pre-flight `docker compose config` validation runs before every write to `~/authentik/docker-compose.yml` and refuses to write on failure.
- [ ] The Sync webadmin button description no longer says "Does not restart anything."
- [ ] Idempotency: a second consecutive Update Now after v0.9.25 makes zero changes to a clean `~/authentik/docker-compose.yml` (no log churn, no recreate).

## Test plan

### Phase 1 — synthetic duplicate-cap_drop install

1. On a sacrificial host, deploy Authentik via the console.
2. Manually inject a second `cap_drop:`/`security_opt:` block into the `server:` service of `~/authentik/docker-compose.yml`, immediately above `command: server`.
3. Confirm `docker compose config` fails with the duplicate-mapping-key error.
4. Click **Update Now** in the console.
5. **Expect:** `_dedupe_authentik_capdrop()` strips the duplicate, validator passes, `worker server ldap` recreate succeeds, log shows the dedupe message.
6. Click **Sync webadmin to Authentik** — expect green success.

### Phase 2 — field validation on `aj.takdfir.com`

1. SSH to the Authentik host, capture `~/authentik/docker-compose.yml` for forensic archive.
2. Trigger Update Now from the console.
3. Verify the dedupe log line appears in the Updates pane.
4. Verify `docker compose -f ~/authentik/docker-compose.yml config` exits 0.
5. Click **Sync webadmin to Authentik** — verify the red banner is gone and 8446 login works with the configured password.
6. Confirm ATAK clients (e.g. `CA-COR-AJ-N20`) still bind and stream as expected — no LDAP-side regression from the LDAP outpost recreate.

### Phase 3 — idempotency

1. On any clean install (fresh or post-dedupe), run Update Now twice in a row.
2. Verify the second run logs `Post-update: Authentik compose already hardened — no changes needed` (the existing branch at `app.py:43652-43653`).
3. Verify `~/authentik/docker-compose.yml` byte-identical between the two runs (no whitespace drift).

### Phase 4 — error-surfacing regression test

1. On a sacrificial host, manually break `~/authentik/docker-compose.yml` (e.g. introduce an unmatched brace).
2. Click **Sync webadmin to Authentik**.
3. Verify the banner shows the *full* parse error (line/column from Compose v2), not a 120-char truncation.
4. Verify the remediation hint appears if the error matches the duplicate-mapping-key heuristic; otherwise it doesn't.
