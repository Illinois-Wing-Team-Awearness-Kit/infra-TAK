## Bug fix — Guard Dog socket leak (console hang after ~3 days)
- **Symptom:** After ~3 days of uptime, port 5001 stops accepting connections; gunicorn accumulates 140+ leaked sockets
- **Root cause:** Five `urllib.request.urlopen()` calls in Guard Dog health checks never called `resp.close()`. Affected: `authentik` service check, `nodered` service check, `remotedb` monitor, `authentik_http` monitor, `nodered_http` monitor
- **Fix:** Wrapped all five with `with urllib.request.urlopen(...) as resp:` — no logic change, sockets released immediately after each check

## Security hardening — CVE-2026-31431 (Copy Fail) — built May 6 2026

### What was built this session

All changes are in `app.py` only. Four edit locations:

**1. Authentik compose template** (~line 22574 in `app.py`)
- Removed `/var/run/docker.sock:/var/run/docker.sock` from worker volumes
- Added `cap_drop: ALL` + `security_opt: no-new-privileges:true` to `server`, `worker`, and `ldap` services
- `postgresql` and `redis` left untouched (infrastructure services)

**2. `_cloudtak_build_override_yml()`** (~line 13889 in `app.py`)
- Added `cap_drop: ALL` + `security_opt: no-new-privileges:true` to `api`, `events`, and `media` services in the generated `docker-compose.override.yml`

**3. TAK Portal install patching** (~line 11526 in `app.py`)
- Added idempotent `cap_drop`/`no-new-privileges` patch after the existing `extra_hosts` block, applied at install time

**4. `_auto_harden_containers()` post-update migration** (~line 35221 in `app.py`)
- New nested function inside `_run_post_update()`, called as part of `Update Now` sequence after `_auto_authentik_ports()` and `_auto_nodered()`
- **Authentik**: reads `~/authentik/docker-compose.yml`, removes `docker.sock` line if present, adds `cap_drop`/`no-new-privileges` to server/worker/ldap via string anchors (`command: server`, `command: worker`, `AUTHENTIK_TOKEN: placeholder`), recreates only changed containers with `docker compose up -d --force-recreate worker server ldap`
- **TAK Portal**: reads `~/TAK-Portal/docker-compose.yml`, adds flags if missing, recreates with `docker compose up -d`
- Both fully idempotent — if already hardened, prints confirmation and skips

### Why the docker.sock had to go
Authentik's upstream quickstart mounts `/var/run/docker.sock` into the worker for Docker-managed outpost deployments. infra-TAK uses the **embedded outpost** (Caddy `forward_auth` → Authentik server) and a standalone `ldap` container — neither touches the socket. With it mounted, a compromised Authentik worker had full Docker daemon access = host root. It was never needed and is now gone from both the template and all live deployments on next update.

### Node-RED hardening (pre-existing, already deployed)
- Image pinned to `nodered/node-red:4.0` (was `:latest`)
- `user: 1000:1000`, `mem_limit: 2g`, `restart: unless-stopped`, `cap_drop: ALL`, `no-new-privileges: true`
- `env_file: .env` for optional adminAuth
- Local deploy: scoped per-file cert mounts; remote deploy: port bound to `127.0.0.1:1880`

## Security — Console service runs as `takwerx` (non-root) — built May 6 2026

### What was done

`app.py` and `start.sh` were updated to run the console as a dedicated `takwerx` system user rather than root. This is a compliance/audit change, not a strong security boundary (the user has `NOPASSWD: ALL` sudo and is in the `docker` group).

**Two-phase rollout (because Python venvs can't be safely relocated from the running process):**

1. **Automatic on "Update Now"** — `_auto_provision_takwerx()` creates the `takwerx` user, sudoers, and groups. Console keeps running as root.
2. **One-time CLI step for /root/-based installs** — console shows a blue banner: "Run `sudo bash start.sh`." Operator SSHes in once, runs it. `start.sh` stops the service, moves install dir to `/home/takwerx/`, rebuilds the venv at the new path, rewrites the service unit, restarts. Banner clears automatically (purely runtime-derived from `os.getuid()`).

Fresh installs run both phases in one `sudo ./start.sh` invocation — no banner ever appears.

**`start.sh`**
- Added `provision_takwerx()` — called during every `start.sh` run after cert generation, before service creation
- Creates `takwerx` user if absent (`useradd -m -s /bin/bash takwerx`)
- Adds to `sudo` and `docker` groups
- Writes `/etc/sudoers.d/takwerx` with `takwerx ALL=(ALL) NOPASSWD: ALL` (440 perms)
- Migrates existing module dirs from `/root/` → `/home/takwerx/` on upgrade: `authentik`, `CloudTAK`, `node-red`, `TAK-Portal`
- `chown -R takwerx:takwerx` on both `/home/takwerx` and `$INSTALL_DIR`
- `takwerx-console.service` now has `User=takwerx` and `Environment=HOME=/home/takwerx`

**`app.py`**
- Three new helper functions added near the top (after rate-limiter globals):
  - `_sudo_wrap(cmd)` — prepends `['sudo', '-n']` when `os.getuid() != 0` and first element isn't already `sudo`
  - `_write_priv(path, content, mode='w')` — writes to privileged paths via `sudo tee` when not root
  - `_read_priv(path)` — reads privileged paths via `sudo cat` when not root
- 86 subprocess calls wrapped with `_sudo_wrap()`: 82 × `systemctl`, 2 × `apt-get`, 2 × `ufw`
- 13 direct `/etc/` file writes converted to `_write_priv()`: systemd units, fstab, postfix configs
- 1 `/etc/ssh/sshd_config` line-iteration read converted to `_read_priv()`
- All 5 `User=root` strings in generated systemd unit templates changed to `User=takwerx` (mediamtx ×2, mediamtx-webeditor ×2, tak-snapshot)
- `_find_settings()` fallback list now includes `/home/takwerx/infra-TAK` (before the legacy `/root/infra-TAK` entry)
- `webadmin-fed.p12` remote copy changed from `/root/` to `$HOME/` in both SSH probe calls and log messages

**6 shell scripts** — all hardcoded `/root/module` primary paths removed, replaced with `${HOME:-/home/takwerx}/module`:
- `scripts/guarddog/tak-post-start.sh` (authentik, CloudTAK, node-red)
- `scripts/guarddog/tak-boot-sequencer.sh` (authentik, CloudTAK, node-red)
- `scripts/guarddog/tak-nodered-watch.sh`
- `scripts/guarddog/tak-cloudtak-watch.sh`
- `scripts/guarddog/tak-authentik-watch.sh`
- `scripts/fix-mediamtx-stream-redirect.sh`

**14 doc files** — `/root/infra-TAK` and `/root/module` references updated throughout `docs/`.

## Node-RED hardening plan — full context

This is the biggest single chunk of work in v0.9.2. Read this section before touching Node-RED on the VPS.

### Why this was a conversation

Colleagues flagged Node-RED as a security flaw because it uses the TAK Server **admin cert** (`admin.pem`/`admin.key`) to call the Mission API. The background question was: is that actually a flaw, or just uncomfortable? Deep answer:

- **Same pattern as TAK Portal.** TAK Portal (the official component) also authenticates to TAK Server using `admin.p12` — the same identity as `admin.pem`. So Node-RED following this pattern is consistent with what ATAKers have already blessed.
- **Real risk is the blast radius.** If Node-RED's runtime is compromised (malicious npm package, supply-chain attack on a contrib node, attacker who got into the editor), they inherit full TAK admin authority and could read/write/delete every DataSync mission on the server. The cert can also be read from the container filesystem and shipped anywhere if egress is unrestricted.
- **The cert itself wasn't the root problem.** The root problem was: (a) no egress filtering (exfil possible), (b) Node-RED container running as root with no capability restrictions, (c) whole `/opt/tak/certs/files` tree mounted instead of just the files it uses, (d) image pinned to `:latest`.

### Why we couldn't just use a scoped cert

In April 2026 a scoped cert (`nodered-global-datasyncfeed`) was tested and failed. TAK Server 5.x has an **unfixed bug**: x509 certificates that resolve their groups via LDAP get **OUT-only** direction on every group, even when the LDAP base group says BOTH. Without IN direction, the Mission API write path (`PUT /Marti/api/missions/.../contents`) is rejected with 403. This blocked every previous attempt to use a non-admin cert for DataSync.

### The new idea — flat-file `nodered` user

The `admin` user is defined in `/opt/tak/UserAuthenticationFile.xml` — a **flat-file** user, not LDAP. Flat-file users declare their groups directly in XML (`<groupList>DATASYNC-FEEDS</groupList>`). They never traverse the LDAP group-resolution path that has the bug.

**Hypothesis:** create a flat-file `nodered` user in UserAuthenticationFile.xml with `ROLE_USER` (not `ROLE_ADMIN`), generate its cert with `makeCert.sh client nodered`, and let Node-RED use that cert. When Node-RED calls `PUT /Marti/api/missions/{name}?creatorUid=nodered`, it becomes the **owner** automatically — no role-elevation hack needed because the creator is always the owner.

If this works: least-privilege cert, real audit attribution (`creatorUid=nodered` in TAK Server logs), blast radius limited to DataSync feeds in the configured group.

If this fails (i.e. flat-file users hit the same group-direction bug as LDAP users): keep `admin.pem`, but harden the runtime around it.

### Phase structure

The work was split into 5 phases. Phases 0 and 2 are the ones that matter tomorrow.

#### Phase 0 — Validation spike (NOT YET RUN — operator must do this on VPS)

Full test plan in `docs/SPIKE-flatfile-nodered.md`. Six curl tests (T0–T6) against a live TAK Server:

| Test | What it proves | Gate |
|---|---|---|
| T0 | `nodered` user loaded by UserManager | sanity |
| **T1** | `GET /Marti/api/groups/all` with nodered.pem returns BOTH IN+OUT on DATASYNC-FEEDS | **CRITICAL — if OUT-only, Phase 1A is dead** |
| T2 | `PUT /Marti/api/missions/spike-test?creatorUid=nodered` returns 200/201 | missions can be created |
| **T3** | Stream CoT to spike-test via TCP 8089, then `PUT /contents` → 200 | **CRITICAL — proves DataSync writes work** |
| T4 | Field LDAP user subscribes → gets MISSION_READONLY_SUBSCRIBER | defaultRole enforced |
| T5 | `nodered.pem` DELETE from mission → 200 | owner can remove |
| T6 | Field user PUT to mission → 403 | read-only enforcement works |

**Decision gate:**
- T1 + T3 both pass → **Phase 1A** (migrate Node-RED to flat-file nodered cert)
- Either fails → **Phase 1B** (keep admin, harden runtime only)

Run the spike by following `docs/SPIKE-flatfile-nodered.md` step-by-step. It includes the exact `openssl rsa` and `curl` commands to copy-paste.

Fully backward-compatible — existing installs with only `admin.pem` continue working unchanged.

**`nodered/build-flows.js` — elevation snippets (both instances):**

#### Phase 1A — Flat-file nodered migration (ONLY if spike passes)

Everything is wired defensively — the code activates only when `/certs/nodered.pem` exists, falls back to admin otherwise. No behavior change on existing installs until operator explicitly runs the bootstrap.

**`scripts/bootstrap-nodered-flatfile.sh` (new):**
Run on the TAK Server host after Phase 0 confirms T1+T3 pass:
```bash
sudo bash scripts/bootstrap-nodered-flatfile.sh apply
```
This:
1. Generates `/opt/tak/certs/files/nodered.{pem,key,p12}` via `makeCert.sh client nodered`
2. Adds `<user identifier="nodered" role="ROLE_USER"><groupList>DATASYNC-FEEDS</groupList></user>` to `/opt/tak/UserAuthenticationFile.xml` using Python XML parser (safe — no regex string surgery on production XML)
3. Restarts TAK Server and waits for `:8089` to come up
4. `chmod 644` the cert files so the Node-RED container (UID 1000) can read them
5. Verifies via `UserManager.jar listusers`

The bootstrap is idempotent — safe to re-run. Use `--check` to inspect state without changes, `--remove` to roll back the XML entry.

After bootstrap: `bash nodered/deploy.sh` — the deploy script sees `nodered.pem` and auto-fills `tls_tak` with it. Go to Node-RED editor → TAK Mission API TLS → enter passphrase `atakatak` → Deploy.

### Critical gotcha — snapshot + Phase 1A rollback

TAK Server snapshots now capture `UserAuthenticationFile.xml` (fixed in this release). If you complete Phase 1A on the VPS (bootstrap adds `<user identifier="nodered">` to that file) and then need to restore a snapshot taken **before** the bootstrap, the XML entry won't be in the snapshot. Certs will survive (in `certs/`), but TAK Server won't recognize `nodered` as a valid user. Fix: re-run `sudo bash scripts/bootstrap-nodered-flatfile.sh apply` after restore — idempotent, re-adds just the XML entry.

The cleanest approach: take a fresh snapshot **after** completing Phase 1A so both the cert files and the XML entry are captured together.

## CVE-2026-31431 ("Copy Fail") — action required on VPS
A Linux kernel vulnerability published 2026-04-29 allows unprivileged container escape to host root. The only real fix is patching the kernel:

```bash
apt update && apt full-upgrade
reboot
```

The reboot triggers Guard Dog's boot sequencer — all containers restart in the correct order automatically.

```bash
cd ~/infra-TAK && bash pull-dev-and-restart.sh
```

Then hit **"Update Now"** in the console UI (or it fires automatically on restart since version changed). Watch the console log for these lines confirming the CVE hardening fired:

```
Post-update: Authentik — removed docker.sock from worker
Post-update: Authentik compose hardened — recreating worker/server/ldap
```

If the compose files were already patched in a prior run, you'll see the idempotent versions instead:

```bash
# Verify Authentik is still healthy after worker/server/ldap recreate
docker ps --filter name=authentik --format "{{.Names}}: {{.Status}}"
```

Expected: all Authentik containers show `Up ... (healthy)`, no `docker.sock` in the worker.
